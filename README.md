# Cross-domain early-warning: one detector, three machines

A single, transparent failure-prediction detector applied **unchanged** to three very
different real-world systems: a jet engine, a lithium-ion battery, and a mechanical
bearing. In each case it watches a *health margin* erode toward a failure level,
projects the *time-to-exhaustion*, and raises a latched early warning **before** the
system fails.

**Live demo:** `https://oren-speiser.github.io/cross-domain-earlywarning/`(interactive: pick a machine, press play)

The point is not any one domain. It is that the same simple law (margin / rate /
time-to-exhaustion) gives reliable advance warning across **thermodynamic**,
**electrochemical**, and **mechanical** failure, covering both gradual degradation
(engine, battery) and abrupt collapse (bearing).

## Results

| Domain | Data | Result |
|---|---|---|
| Turbofan engine | NASA C-MAPSS FD001 (simulation benchmark) | 99/100 engines warned with positive lead; median **16 cycles** advance warning; alarm fires at ~92% of life |
| Li-ion battery | NASA Li-ion aging (**real measured**) | **18 to 42** discharge cycles of advance warning (cells B0005 / B0006 / B0018) |
| Bearing | FEMTO PRONOSTIA (**real measured**) | abrupt failure mode; still flagged **2 to 5 minutes** before the bearing lets go |

Every number above is regenerated by the analysis scripts from the public data they
download.

## What is and is not claimed (honest scope)

- The engine dataset (C-MAPSS) is NASA's standard turbofan prognostics benchmark, but it
  is **simulation-generated**, not physical-engine telemetry. The battery and bearing
  datasets are **real measured** run-to-failure data.
- The scalar health margin for the engine and bearing is constructed transparently from
  the raw sensors (see each script). The failure threshold is the observed degradation
  level at failure (analogous to a known end-of-life spec); this is in-sample and
  disclosed.
- The demo shows the **result** of the detector. The detector here is the public
  time-to-exhaustion method in `detector.py`: deliberately simple and fully reproducible.
- Bearings fail abruptly, so the alarm fires late in life with a short (but real) lead.
  This is reported as-is, not tuned to look earlier.

## The detector

`detector.py` is the same code in all three domains:

1. smooth the health margin (causal median + short moving average),
2. estimate the local loss rate (trailing-window linear-fit slope),
3. `ttx = remaining margin / loss rate`,
4. raise a **latched** warning when `ttx` stays below a horizon for several readings.

Only two things change per domain: how the health margin is built from that domain's
raw sensors, and the smoothing windows, scaled to each domain's sampling interval.

## Reproduce

```bash
pip install -r requirements.txt
python engine.py     # downloads C-MAPSS FD001, prints stats, writes engine.json + figure
python battery.py    # downloads NASA Li-ion .mat files, writes battery.json
python bearing.py    # sparse-downloads 3 FEMTO bearings, writes bearing.json
python build.py      # rebuilds index.html from the processed data
```

The raw datasets are downloaded from their original public homes (below) into a local
`raw_data/` folder. Only the small processed series (`*.json`) are committed, so the
demo runs immediately and every number is inspectable without re-downloading gigabytes.

## Data sources

- **NASA C-MAPSS** turbofan degradation (Saxena & Goebel, 2008). NASA Prognostics
  Center of Excellence, https://data.nasa.gov . Download mirror used:
  `hankroark/Turbofan-Engine-Degradation`.
- **NASA Li-ion** battery aging (Saha & Goebel, 2007). NASA PCoE, https://data.nasa.gov .
  Download mirror used: `XiuzeZhou/NASA`.
- **FEMTO PRONOSTIA** bearing run-to-failure (Nectoux et al., IEEE PHM 2012 Prognostic
  Challenge). FEMTO-ST Institute. Mirror used:
  `wkzs111/phm-ieee-2012-data-challenge-dataset`.
